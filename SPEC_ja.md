Flat Markdownは、通常のMarkdown文書をアウトライナーの項目(item)として扱うための明確なルールを定めたものである。

# Flat Markdownのポリシー

- アウトライナーの項目の区切りが視覚的に判別しやすく、テキストエディタで編集しやすいことを目指す。
- 一般的なGitHub Flavored Markdownで書かれた文書を全て受け入れる。
- 拡張されたMarkdown記法を持つ。
- 改行について、独自の解釈を行う。

# Flat Markdownが扱う文書（ページ）の構造

```
page
├── page-metadata[]
└── item[]
    ├── body
    └── item-metadata[]
```

- **ページ (page)**
  - ページのメタデータ（page-metadata）のリストと、項目（item）のリストで構成される。
  - ページのメタデータ
    - 冒頭の`---`で挟まれた領域は YAML front matterとして解釈し、ページのメタデータとして扱う。
  - 項目
    - アウトライナーの項目。Markdownで記述される。

- **項目 (item)**
  - 本文（body）と項目のメタデータ（item-metadata）のリストで構成される。
  - 本文
    - Markdownで記述される。
  - 項目のメタデータ
    - `<!-- HTMLのコメント -->`形式で記述される。


## 空行による項目の区切り

- Markdown文書中の1つ以上の空行を、項目の区切りとして解釈する。
  - 基本的には、Markdownにおける段落が、項目に該当する。
- 連続する複数の空行も、1つの区切りとして扱う。
- 項目の本文内の空行について認めない。ただし、別途定めるcode fence内の空行のみを例外とする。
- 文書末尾に1つ以上の空行がある場合、無視される。
- front matterの後の最初の空行はfront matterの終端であり、項目の区切りではない。

## 項目

各項目は、本文として任意のGFMを含むことができる。
 
### GFMとの違い

#### 空行の扱い

- 空行の段落は、`<br />`のみ記述された行によって表現する。
- 項目内の意図的な空行について
  - code fenceで囲まれた内部の空行は、空行として扱う。これは、空行による項目の区切りとして扱われない。
  - それ以外の意図的な空行は、視覚的に判別できるよう、必ず`<br />`のみの行によって表現する。
- パーサーは、code fence外のUnicode White_Spaceプロパティを持つ文字のみを含む行（空行・スペースのみの行・全角スペースのみの行など）を自動的に`<br />`行として解釈する。
  - 空白は、UnicodeでWhite_Spaceプロパティを持つ文字すべて。全角スペース（U+3000）なども含まれる。

#### 改行の扱い

- 全ての改行コードを、hard line break （`<br />`）として扱う。
- GFMのHard line breaks記法（行末の2つ以上の空白、またはバックスラッシュ）は無視される。

#### 拡張構文
- footnote
- alert
- subtext
- highlight
- superscript
- subscript
- spoilered_text
- shortcode
- math
- hashtag
- underline
- wikilink

### 項目のメタデータ

項目のうち、本文の最後の改行以降、空行またはページ末までの行は、メタデータ部とする。

#### メタデータの種類
メタデータは、大きく分けて、IDとプロパティの2種類がある。

- ID
  - 必須のメタデータ。データの相互参照や同期のために用いる。
- プロパティ
  - key=value形式のプロパティ。項目の折りたたみ状態（collapsed）や、タスクの詳細、その他、任意のkeyとvalueの組。

アウトラインにおける項目のインデントは、項目のプロパティとして記述する。
- デフォルトの深さをレベル1として数え、ひとつ右にインデントした場合は、`<!-- level=2 -->` と書く。
- レベル1の場合はデフォルトなのでlevelプロパティを省略できる。
- つまり、一般的なMarkdownをFlat Markdownのルールで解釈すると、levelプロパティがないため、全ての項目はレベル１の深さ（インデントなし）となる。

#### メタデータの記法

- 1種類のメタデータにつき1行ずつ記述する。
- メタデータはHTMLのコメント文 <!-- --> の中に記述される。これらは、HTMLに変換された際に表示されることはない。
- メタデータとして有効なHTMLコメントは、行頭から始まる1行のコメント文で、かつ、`<!-- key=value -->`のプロパティパターンまたは`<!-- HMTID -->`のパターンにマッチするもののみである。これに該当しないHTMLコメントは本文として扱う。
- HMTID
  - https://github.com/sosuisen/hmtid
  - ハイフン区切りかつ日時も区切る形式の27字のID。
- メタデータは手書きでも書くことはできるが、GUIを用いた入力を想定している。

### 補足
- 空行に関する例外を減らすため、multiline blockquote（`>>>`）のような拡張ブロックは認めない。

# Flat Markdownの特徴

- 視覚的な判別性
	- 空行で区切られた項目は、視覚的にすぐ区別することができる。
- 互換性
	- ブログ記事のような普通のページを書いたMarkdown文書もアウトライナーで扱うことができる。
	- 一般的なMarkdownなので、あらゆるMarkdownツールで編集・表示できる。
- データ変換の容易さ
	- Markdownデータを元にWebページその他へパブリッシュする多数のツールに対応できる。
	- 項目を分割するコードを簡単に書くことができる。
	- アウトライン項目をMarkdownのリスト木で表現するのが好みであれば、Flat Markdownを解釈してMarkdownのリスト木に変換するツールは、ものすごく簡単に作れる。公式にも提供する予定だが、現在のAIコーディングの力を借りるなら、自力でも3分とかからない。
- インデントなしの哲学
	- アウトライナーの木構造をMarkdownで表現するアイデアとしては、Markdownのリスト木がまず思い浮かぶだろう。しかし、それは採用しなかった。
	- 手書きで項目を編集するときに、リスト木は先頭の空白の数を間違えやすく、全ての行におなじ数の空白を入れる手間がある。しかも、インデントが多いと、スマートフォンのような幅の狭い画面では、画面の外に項目が出てしまうのが不便だった。
	- Markdownは汎用マークアップ言語であり、アウトライナーの項目を表現するのに必ずしも向いている記法ではない。しかし、Markdwonには現代における中間言語として価値がある。
	- インデントをなくすことで、これらの問題は全て解決される。
	- インデントをなくしたことで、新たな利点も生まれた。
		- まず、インデントを変更したときの文書の差分が小さい。変更されるのはlevelプロパティ行のみである。
		- 項目にメタデータ部を追加する場合、Markdownのリスト木はかなり混乱した見た目になる。しかし、空行区切りを採用すると、メタデータ部を追加しても項目の区切りがくっきりと視覚的に判別できるようになった。
	- 背景としてはRoam ResearchやLogseqのような名前空間とページ単位を持つアウトライナーを念頭に置いている。
		- ページが階層をもっている分、Workflowyのような１つの大きなツリーを持つアウトライナーと比べ、インデントが深くならない。
		- インデントが浅いなら、Markdown上で視覚的にインデントがない場合の問題は軽減される。
			
# PetaJournalにおけるFlat Markdownの運用

## メタデータ

メタデータの順序は仕様としては定めないが、journal-serverでは次の順序でシリアライズする。
	- ID
	- インデントレベル
	- その他のプロパティ

## Task Listの解釈

- GFMのタスクリスト `- [ ] ` から始まる項目については、PetaJournalでは項目全体をタスクとして扱い、バレットをチェックボックスで置き換える。

## 埋め込み

画像を始めとするメディアの埋め込みフォーマット
```
![alt](img.png#w=300&h=240)
![alt](video.mp4#w=640&autoplay)
![alt](https://youtube.com/watch?v=xxx#w=640&start=30)
![alt](doc.pdf#page=3&w=800)
```
サーバ側のパーサは、ASTにembedノードを追加して返す。どうレンダリングするかはフロントエンド側の仕事。

## タグ

- 項目の本文中のどこにでも置くことができる。 `#タグ名`
- ページのタグはFront Matterのプロパティとして設定。
- 以下の文字のみ有効
	- Unicode英数字
		- Rustの char::is_alphanumeric() に基づく
		- 漢字: タグ, 日記 — Letter (Lo) → 含まれる
		- ひらがな: たぐ — Letter (Lo) → 含まれる
		- カタカナ: タグ — Letter (Lo) → 含まれる
		- 全角数字: １２３ — Digit (Nd) → 含まれる
		- 半角英数字: abc, 123 — 含まれる
		- 句読点は含まれない。
	- 記号３つ `-` `_` `/`
	- `/`がタグ名の先頭または末尾に来るのは不許可
- タグの開始条件
	- 文頭または、半角スペースに続く#
		- ==Logseqは半角スペース不要なので、ここは互換性がないため注意==
	- バックスラッシュでエスケープできる。`\#`
- タグの終了条件
	-  許可文字以外の文字で終了とする、というだけのルールでOK。
		- `#mytag。次`の場合、タグはmytag。
		- 日本語句読点、(), [] 等で自然に切れる。
		- 空白文字や文字列末尾でも切れることになる。
- タグの意味
	- 動作はLogseqと同じ。つまり、タグ名はWikilinkと同等であり、リンク先はページとなる。

## Markdownファイルにおける扱い

変換ルール `タグ <---> Markdownリンク` 

- リンク解決ONの場合
  - `#タグ名 <---> [#タグ名](../タグ名.md)`
- リンク解決OFFの場合
  - 変換なし

# Wikiリンク記法

- ページへのリンク
	- Wikiリンク記法 `[[ページ名のフルパス]]`
	- ラベルを指定する場合 `[[ページ名のフルパス|ラベル]]`
- ブロックへのリンク
	- `[[ページ名のフルパス#id]]`
	- ラベルを指定する場合 `[[ページ名のフルパス#id|ラベル]]`

## Markdownファイルにおける扱い

- 変換ルール `Wikiリンク <---> Markdownリンク` 

- リンク解決ONの場合
  - Markdownへ書き出しする際は、WikiリンクではなくMarkdownリンクに変換して保存する。
  - 一般のMarkdownツールではWikiリンクを扱えないので。
  - つまり、アプリで書くときはWikiリンクで、シリアライズされるデータはMarkdownリンクにする。
  - `[[ページ名のフルパス]] <---> [ページ名のフルパス](../ページ名のフルパス.md)`
  - `[[ページ名のフルパス|ラベル]] <---> [ラベル](../ページ名のフルパス.md)`
- リンク解決OFFの場合
  - 変換なし

## パスの扱い
- Obsidianは、他に同じファイル名がない場合、フォルダパスを省略できる。
- Logseqと同じフルパスのほうがシンプルでいいと思う。
	- よって、フォルダが変わったら、リンクは全て書き換え。
	- このほうが、WikiリンクとMarkdownリンクの変換も容易。
